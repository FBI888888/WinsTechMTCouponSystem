# WinsTech MT Coupon System 性能优化方案（修订版）

> 适用场景：千万级数据量、桌面端 + 后端服务协同、外部平台 API 高频交互
> 本文档基于当前项目代码实现重写，不是通用模板。

---

## 一、文档目标

本方案用于回答三个问题：

1. 当前项目的真实性能瓶颈在哪里
2. 旧版优化方案哪些方向正确，哪些建议不能直接落地
3. 如果要面向千万级数据和企业级部署，应该先做什么、后做什么

本文优先解决以下路径：

- 前端与后端交互
- 后端与数据库交互
- 后端与美团外部接口交互
- 定时扫描链路

---

## 二、项目现状与关键判断

### 2.1 当前架构形态

当前项目不是纯 Web 应用，也不是纯后端集中式架构，而是一个混合架构：

- 前端是 Electron + React
- 后端是 FastAPI + SQLAlchemy + MySQL
- 前端一部分能力走后端 API
- 前端另一部分能力直接通过 Electron 调本地能力和外部接口

这意味着性能问题不能只看数据库查询，还必须同时看：

- 前端是否把大数据量先拉到本地再处理
- 后端是否把外部 API 调用放在串行链路上
- 同一业务动作是否重复经过前端、本地 Electron、后端三层

### 2.2 核心结论

基于当前代码，项目的主要瓶颈不是单点，而是四类问题叠加：

1. 数据流设计存在“先全量拉取，再本地过滤”的路径
2. 券码查询以“券码”为单位驱动外部 API，导致同一订单重复请求
3. 定时扫描和手动扫描整体偏串行，吞吐上限偏低
4. 列表、统计、日志接口还没有切换到适合大数据量的查询模型

结论上，旧版文档的方向并非全部错误，但优先级排序和部分实现建议需要重写。

## 三、已确认的性能热点

### 3.1 订单列表接口

当前订单列表接口存在以下问题：

- 每次列表请求都执行 `count()`
- 使用 `offset + limit` 分页
- 搜索使用 `%keyword%` 模糊匹配
- 状态筛选部分依赖 `showstatus like '%xxx%'`

这四点同时存在时，千万级数据下压力不只是 `count(*)`，而是：

- `count(*)` 本身昂贵
- 深分页性能持续下降
- 文本模糊匹配难以利用普通索引
- 文本状态筛选使复合索引收益变差

结论：

- 旧文档把问题主要归因到 `count()`，判断不完整
- 订单列表应该整体重构查询模型，而不是只给 `count()` 加缓存

### 3.2 订单同步链路

当前订单同步流程是：

1. 前端先调用 `/api/orders/ids` 拉取某账号全部 `order_id`
2. 把这些 ID 放到前端内存中
3. 再从 Electron/外部接口拿订单
4. 在前端本地做去重
5. 再把新增订单批量回传后端保存

这个模型在数据量上来后会出现明显问题：

- 后端返回全量 `order_id`，接口本身会越来越重
- 前端需要维护越来越大的 `Set`
- 网络和序列化成本持续增长
- 真正的幂等控制落在前端，不适合企业级系统

结论：

- 这条链路的第一优先级不是换 `bulk_insert_mappings`
- 第一优先级是把“去重与幂等”收回后端

### 3.3 待查询券码订单接口

当前 `/api/orders/pending-coupon-query` 会：

- 把所有待查券码订单一次性 `all()` 取出
- 前端再限制本次最多处理 1000 条

问题在于：

- 数据库已经完成了全量扫描和全量返回
- 后端已经承担了最大成本
- 前端的“只扫 1000 条”并没有减少后端代价

结论：

- 这个接口必须改为服务端限流
- 应该在后端支持 `limit`、游标或批次领取

### 3.4 券码查询链路

当前券码查询存在两个层面的重复工作。

#### 第一层：后端券码查库

`batch_find_coupons_by_codes()` 目前仍然是循环调用 `find_coupon_by_code()`，属于典型 N+1。

这一点旧文档判断是对的。

#### 第二层：外部 API 调用粒度错误

更严重的问题是，`/api/coupons/query-backend` 虽然先批量查到了券码和订单，但后续仍然按“输入券码”逐条调用美团接口。

这会导致：

- 同一订单下多个券码，重复调用同一个外部 API
- 重复统计 `CouponHistory.count()`
- 重复查询该订单下全部券码
- 外部 API 压力和接口耗时被放大

结论：

- 真正需要的不是“只修一个 N+1”
- 而是把券码查询流程改为“按订单聚合，再分发到券码结果”

### 3.5 定时扫描与手动扫描链路

当前扫描服务的执行模型大致是：

1. 检查账号有效性
2. 拉取订单列表
3. 过滤未落库订单
4. 对每个订单启动一次 Node 子进程查询券码
5. 串行保存订单和券码
6. 账号之间和订单之间主要靠 sleep 做节流

存在的问题：

- 整体吞吐偏串行
- 每单一次 `subprocess.run` 成本较高
- `httpx.AsyncClient` 每次请求都重新创建
- 对外部 API 的重试、限流、并发控制还比较粗放

结论：

- 这条链路是企业级场景下的核心吞吐瓶颈之一
- 旧文档对它关注不够，只提了分布式锁，不够

### 3.6 Dashboard 统计接口

首页统计接口当前会执行多次独立 `count()` 查询。

在数据量小时问题不大，但企业场景下会带来持续压力，尤其当：

- Dashboard 是高频入口
- 统计口径还包含 `like '%待消费%'` 这类文本筛选

结论：

- 统计接口应纳入优化范围
- 不能只盯业务列表页

## 四、旧版方案中需要修正的地方

### 4.1 可以保留的方向

以下方向仍然成立：

- 修复券码查询 N+1
- 为高频查询补充复合索引
- 引入 Redis 作为共享缓存和分布式能力基础
- 增加慢请求监控
- 长期考虑归档、读写分离

### 4.2 不能直接照抄的建议

#### 问题一：用户缓存示例会破坏鉴权对象类型

旧版示例建议把用户缓存成字典并直接返回。

但当前鉴权链路后续会使用：

- `cached_user.is_active`
- `current_user.role`

如果缓存返回的是字典，代码会直接出错。

正确做法应为二选一：

- 继续缓存 ORM 对象，但只作为单进程短缓存
- 或缓存精简字段，命中后重新构造鉴权上下文对象，不直接冒充 `User`

#### 问题二：Axios 请求去重示例无效

旧文档中给 `config` 塞 `_pendingPromise` 的方式，并不能让 axios 自动复用请求。

正确实现通常有两种：

- 在 API 封装层自己维护 `Map<key, Promise>`，直接返回已有 Promise
- 或取消旧请求，只保留最新请求

单靠拦截器里返回一个额外字段，不会真正去重。

#### 问题三：Redis 锁示例不是分布式锁

旧文档使用 `cache.set(key, value, ttl)` 作为“抢锁”条件。

如果底层只是普通 `setex`，多个实例都能写入成功，不具备互斥语义。

正确做法：

- Redis 使用 `SET key value NX EX ttl`
- 释放锁时校验 value，避免误删别人的锁
- 生产环境建议使用成熟锁实现或 Lua 脚本

#### 问题四：只给 `count()` 做内存缓存不够

旧文档把订单列表优化集中在 `count()` 缓存上。

这在当前系统中收益有限，因为：

- 多 worker 下缓存不共享
- 搜索条件变化快，命中率不一定高
- 真正的大头还包括深分页、模糊搜索、文本状态筛选

结论：

- `count` 缓存可以做，但不是主方案
- 主方案应是查询模型重构

#### 问题五：过早强调 `bulk_insert_mappings`

批量插入映射比 `bulk_save_objects` 更快，这个判断通常没错。

但对当前项目来说，它不是订单同步链路的第一瓶颈。当前更大的问题是：

- 前端先拉全量 ID 去重
- 去重逻辑放在前端
- 后端没有把幂等写入作为中心能力

结论：

- 可以做，但应排在后端幂等化之后

## 五、修订后的优化优先级

### 阶段 P0：先做结构性止损（最高优先级，1-2 周）

这一阶段不追求“技术上更先进”，只追求把最重的错误路径改掉。

#### P0.1 订单同步改为后端幂等写入

目标：

- 不再让前端拉全量 `order_id` 做去重
- 把去重责任收回后端

建议方案：

- 订单表确保 `(account_id, order_id)` 唯一
- `save-batch` 改为真正的幂等写入接口
- 前端直接上传本次同步结果，不再先取全量 ID
- 后端用批量 upsert 或按唯一键冲突处理

预期收益：

- 显著减少网络传输量
- 减少前端内存占用
- 同步逻辑更适合企业部署

#### P0.2 `/pending-coupon-query` 改为服务端分批领取

目标：

- 避免后端一次性返回所有待查订单

建议方案：

- 接口增加 `limit`
- 默认只返回固定批次，比如 100/200/500
- 只返回真正需要的字段
- 如后续有多实例消费需求，再引入“领取中”状态或任务表

预期收益：

- 立即降低数据库与接口压力
- 前端也更容易做批处理进度控制

#### P0.3 券码查询按订单聚合调用外部 API

目标：

- 让同一订单只调用一次美团接口

建议方案：

- 先把输入券码映射到订单
- 按 `order_id` 分组
- 每个订单只请求一次外部接口
- 响应结果再映射回多个输入券码

预期收益：

- 外部 API 请求量显著下降
- 后端总耗时大幅下降
- 订单下多券码场景收益最明显

#### P0.4 修复 `batch_find_coupons_by_codes()` 的真实批量查询

目标：

- 把当前循环查库改成真正批量

建议方案：

- 当前券码表 `Coupon.coupon_code in (...)` 一次查出
- 历史表 `CouponHistory.old_coupon_code in (...)` 一次查出
- 批量加载当前券码记录
- 在内存中完成映射组装

预期收益：

- 直接降低查库次数
- 为订单聚合查询打基础

---

### 阶段 P1：重构高频查询模型（1-2 周）

#### P1.1 订单列表改造

目标：

- 让订单列表在千万级数据下仍然可用

建议方案：

- 保留分页，但逐步从 `offset pagination` 迁移到 `keyset pagination`
- 搜索场景和普通列表场景分开处理
- 将 `showstatus` 文本状态逐步收敛为标准化字段

为什么必须做：

- 只加索引，无法根治深分页
- `%keyword%` 和 `%状态文本%` 对索引极不友好

#### P1.2 增加面向真实查询路径的索引

现有模型已具备：

- `orders(account_id, order_id)`
- `coupons(order_id, coupon_code)`
- `coupon_history(order_id, changed_at)`
- `coupon_history(old_coupon_code, new_coupon_code)`

建议新增或评估以下索引：

```sql
ALTER TABLE orders
ADD INDEX idx_orders_account_paytime (account_id, order_pay_time DESC, id DESC);

ALTER TABLE orders
ADD INDEX idx_orders_account_coupon_status_paytime (account_id, coupon_query_status, order_pay_time DESC, id DESC);

ALTER TABLE coupons
ADD INDEX idx_coupons_account_query_time (account_id, query_time DESC, id DESC);

ALTER TABLE coupon_history
ADD INDEX idx_coupon_history_old_code_time (old_coupon_code, changed_at DESC);
```

说明：

- `showstatus like '%xxx%'` 这类条件本身不适合普通索引
- 真正要优化这类查询，核心不是继续补索引，而是状态字段标准化

#### P1.3 Dashboard 统计做缓存或汇总

建议方案：

- 首页统计不直接实时跑多条 `count()` 查询
- 用短 TTL 缓存，或维护汇总表
- 管理后台允许“最近更新时间”展示，而不是强行每次实时精确

---

### 阶段 P2：提升扫描吞吐（2-4 周）

#### P2.1 扫描服务复用网络客户端

当前问题：

- `httpx.AsyncClient()` 每次请求新建

建议方案：

- 为美团订单接口维护可复用客户端
- 在应用生命周期内创建和复用

预期收益：

- 减少连接建立成本
- 稳定长时间扫描任务

#### P2.2 控制并发，而不是纯串行

建议方案：

- 账号之间允许有限并发
- 订单之间允许有限并发
- 并发上限由配置控制
- 区分“正常限流”和“风控保护”

注意：

- 这里不是越并发越高越好
- 要在吞吐与风控之间找平衡

#### P2.3 Node 调用改为常驻 worker

当前问题：

- 每个订单一次 `subprocess.run`

建议方案：

- 改为常驻 Node 进程
- Python 通过 IPC、HTTP 或队列与其通信
- 将签名能力和接口调用能力做成长期运行服务

预期收益：

- 降低进程启动开销
- 提高扫描稳定性
- 更适合后续多账号并发

#### P2.4 将扫描任务抽象为任务队列

建议方案：

- 手动扫描和定时扫描都产生命令型任务
- 后台 worker 消费任务
- API 只负责创建任务和查询状态

这个阶段可以暂不微服务化，但应先把执行模型和 HTTP 请求解耦。

---

### 阶段 P3：共享能力与基础设施（按需推进）

#### P3.1 Redis 的正确引入方式

Redis 应用于以下场景是合理的：

- 多 worker 共享短期缓存
- Dashboard 统计缓存
- 分布式锁
- 简单任务状态缓存

不建议把 Redis 当成“先上了再说”的默认步骤。

引入条件：

- 已经确认需要多进程共享状态
- 已经有明确 key 设计和失效策略
- 已经规划好故障降级方案

#### P3.2 分布式锁

当且仅当出现以下情况时再引入：

- 多实例部署
- 同一个定时任务可能在多个实例上重复执行

建议方案：

- Redis `SET NX EX`
- 锁值校验后删除
- 锁超时必须覆盖任务最长执行时长

#### P3.3 读写分离

适用前提：

- 已有明确读压力瓶颈
- 统计和列表请求远高于写入请求
- 能接受一定复制延迟

当前阶段不建议提前做。

#### P3.4 历史归档与冷热分层

当订单和券码数据持续增长后，归档将是必选项。

建议策略：

- 热数据保留近 6-12 个月
- 历史数据转归档表或归档库
- 管理后台默认查热数据
- 高级查询再跨热冷数据

这比“所有数据永远放主表”更可持续。

## 七、监控与验证方式

### 7.1 必须补齐的观测项

至少记录以下指标：

- 订单列表接口耗时
- 待查订单接口耗时与返回条数
- 单次券码查询涉及的外部 API 调用次数
- 定时扫描单账号耗时
- 定时扫描每分钟处理订单数
- Dashboard 接口耗时
- 数据库慢查询日志

### 7.2 优化验证原则

每一轮优化必须至少回答以下问题：

1. 请求次数是否减少
2. 单次请求返回数据量是否减少
3. 数据库扫描行数是否减少
4. 外部 API 调用次数是否减少
5. 是否引入了新的数据一致性风险

### 7.3 建议的验证动作

```bash
# 1. 订单列表 EXPLAIN
EXPLAIN SELECT * FROM orders
WHERE account_id = 1
ORDER BY order_pay_time DESC
LIMIT 50;

# 2. 待查订单接口压测
wrk -t4 -c20 -d30s http://localhost:8000/api/orders/pending-coupon-query?account_id=1

# 3. Dashboard 接口压测
wrk -t4 -c20 -d30s http://localhost:8000/api/stats/dashboard
```

同时建议在代码中记录：

- 单次券码请求关联了多少输入券码
- 单次券码请求最终调用了多少次美团接口

这是判断“按订单聚合是否生效”的关键。

## 八、风险与边界

### 8.1 当前不建议立即做的事情

以下事项不是不能做，而是不应抢在 P0/P1 前面：

- 读写分离
- 微服务拆分
- 大规模缓存体系
- 全量消息队列化

原因很简单：

- 当前主要瓶颈还在数据流和调用粒度
- 先上复杂基础设施，收益不一定大，复杂度一定会先上升

### 8.2 需要特别注意的一致性问题

- 订单同步必须后端幂等
- 券码更新必须以订单为边界处理事务
- 分布式锁必须确保互斥语义真实成立
- 缓存不能直接替代权限校验对象

---

## 九、最终结论

重写后的判断如下：

1. 旧版方案可以保留“修复 N+1、补索引、引入监控、长期考虑 Redis/归档”的总体方向
2. 旧版方案低估了当前项目真正的结构性瓶颈，尤其是：
   - 前端全量去重
   - 待查订单全量返回
   - 券码查询未按订单聚合
   - 扫描链路串行且高外部调用成本
3. 对当前项目最有价值的优化，不是先换缓存或连接池，而是先改数据流和调用粒度
